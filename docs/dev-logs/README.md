# 开发日志规范

## 📝 文件命名约定

### 格式：`YYYYMMDD-N.md`

- `YYYY`: 四位年份
- `MM`: 两位月份 (01-12)
- `DD`: 两位日期 (01-31)
- `N`: 当天的序号 (1, 2, 3...)

### 示例
- `20250831-1.md` - 2025年8月31日第1次开发记录
- `20250831-2.md` - 2025年8月31日第2次开发记录
- `20250901-1.md` - 2025年9月1日第1次开发记录

## 📋 日志内容模板

每个开发日志应包含以下部分：

### 基本信息
```markdown
# 开发日志 - YYYY-MM-DD 第N次

**开发日期**: YYYY-MM-DD  
**开发时间**: HH:MM - HH:MM  
**开发者**: 人类/AI  
**项目状态**: 阶段X - 功能描述  
**Git提交**: commit-hash  
```

### 核心内容
```markdown
## 🎯 本次目标
- [ ] 目标1
- [ ] 目标2

## ✅ 完成功能
- [x] 已完成功能1
- [x] 已完成功能2

## 🔧 技术实现
### 关键代码变更
- 文件路径: 变更描述

### 技术决策
- 决策1: 原因和考虑

## ⚠️ 遇到问题
### 问题1
- 问题描述
- 解决方案
- 相关文件

## 📝 开发笔记
- 重要发现
- 需要记住的要点
- 下次开发注意事项

## 🚀 下次任务
- [ ] 下次要做的任务1
- [ ] 下次要做的任务2
```

## 🔄 使用流程

### 开始开发时
1. 创建新的日志文件 `YYYYMMDD-N.md`
2. 填写基本信息和本次目标
3. 在开发过程中记录关键信息

### 开发过程中  
1. 记录重要的技术决策
2. 记录遇到的问题和解决方案
3. 记录关键代码变更

### 结束开发时
1. 总结完成的功能
2. 记录未解决的问题
3. 制定下次开发计划
4. 提交git并记录commit hash

## 💡 最佳实践

### 写作要点
- **具体明确**: 描述要具体，避免模糊表述
- **问题导向**: 重点记录问题和解决方案
- **持续更新**: 开发过程中持续更新，不要等到最后
- **关联引用**: 记录相关文件路径和代码行数

### 协作考虑
- **人类可读**: 考虑人类开发者的阅读体验
- **AI友好**: 为AI助手提供清晰的上下文信息
- **搜索友好**: 使用清晰的标题和关键词

### 版本控制
- 每次开发结束后提交日志文件
- 日志文件不要修改，只能新增
- 如需更正，在新日志中说明

## 🎯 目标

通过规范的开发日志：
- 避免人类和AI遗忘重要信息
- 保持项目开发连续性
- 积累技术决策历史
- 便于问题追踪和复盘
- 提升开发效率
